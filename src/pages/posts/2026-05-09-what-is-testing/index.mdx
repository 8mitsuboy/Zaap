# AIに指示が出せなかった自分が、テスト設計を学んで変わったこと

## テストを学ぼうと思ったきっかけ

### 物量勉強法の限界を直感した

新卒で最初に入った案件では、触ったことのない技術や領域を実務レベルで一気に身につける必要があり、かなり挫折しかけていました。

そのとき、これまでみたいに物量で押し切る勉強法では無理だと感じました。だから個別の技術を一つずつ追うより、まずはいろんな技術に共通する幹を掴んだほうがいいと思うようになりました。

その共通部分として自分が気になったのが、テストでした。

### AIに指示が出せなかった

個人で勉強したり開発したりしているときはそれなりにコードが書けるのに、業務になると急に何を実装すればいいのか分からなくなることがよくありました。

だったらAIに指示を出そう、とも思ったのですが、そもそも自分の中で要求が整理できていないので、うまく指示も出せませんでした。

今振り返ると、**実装対象がどう振る舞うべきかを自分で掴めていなかった**のだと思います。だったら、その振る舞いを整理する助けになるテストをちゃんと学んでみよう。そんな流れでテストに興味を持ちました。

---

## このスライドのゴール

> **「AIに指示が出せない・何を実装すればいいか分からない」状態を解消する**

テストの考え方を身につけることで、実装対象の振る舞いを整理できるようになる。
そのゴールに向けて、JSTQBの第1章・第4章の内容を自分の言葉で整理し直します。

## テストの目的

まず大前提として、テストの目的は1つではありません。

ISTQBでは以下のように複数定義されています。

| テストの目的 | 内容 |
|---|---|
| 欠陥の検出 | 故障を引き起こし、ソフトウェアの欠陥を発見する |
| 要件の確認 | 実装が要件・仕様を満たしていることを確認する |
| カバレッジの確保 | 重要な機能や特性を見逃さないようにする |
| リスクの低減 | 本番障害のリスクを事前に下げる |

重要なのは、**目的はコンテキストによって変わる**という点です。

```
納期が迫っている → 必要最低限の機能が動くことの確認が目的
リリース前の決済系 → 欠陥の検出・要件確認が目的
機能追加後のリグレッション → 既存機能への影響がないことの確認が目的
```

この「テストの目的」はテスト計画フェーズで定義されます。
目的が曖昧なままテストを始めると、**終わりのないテストの始まり**になります。

---

## テストプロセスの全体像

テストはいきなりテストケースを書くものではなく、以下のプロセスを踏みます。

```
テスト計画
  └─ テスト目的・リスクを定義

テスト分析
  └─「何をテストするか」を明らかにする
  └─ テスト条件の抽出・カバレッジ基準の設定

テスト設計
  └─「どのようにテストするか」を明らかにする
  └─ カバレッジアイテムの導出・テストケースの設計

テスト実装
  └─ テストデータ・環境を準備する

テスト実行
  └─ テストを実行し、結果を記録する
```

今日の話は主に**テスト分析・テスト設計**にフォーカスします。

## テスト分析：「何をテストするか」

### テスト条件とは

> 「テストによって確認できるもの/こと」

ポイントは**抽象度に幅がある**こと。

| 抽象度 | 例 |
|---|---|
| ハイレベル | 「送料計算が正しく機能すること」 |
| 中レベル | 「会員ステータスによって送料が変わること」 |
| 詳細 | 「一般会員・5000円未満・クレカ支払い時、送料が500円となること」 |

### カバレッジ基準とは

> 「何をもって100%とするかのルール」

テスト条件が決まっただけでは、どこまでやれば完了かが決まりません。
カバレッジ基準を設定することで初めて「完了」が定義できます。

```
カバレッジ基準の例
  ├─「会員ステータスの全パターンを網羅する」
  ├─「境界値は必ずカバーする」
  └─「全属性の全組み合わせを網羅する」
```

カバレッジ基準はリスクと連動します。

| リスク | カバレッジ基準 |
|---|---|
| 高 | 全組み合わせ100% + 境界値必須 |
| 中 | 主要な組み合わせ + 境界値推奨 |
| 低 | イーチチョイス（各パターンが最低1回） |

### テスト条件の抽象度が大事な理由

具体的すぎるテスト条件が渡ってきた場合にリスクがあります。

```
❌「一般会員の場合、正しく送料計算されること」
  → 会員ステータスの観点しか見えない
  → 購入金額・支払い方法の観点が抜ける危険

✅「送料計算が正しく機能すること」
  → 送料に影響する属性を自ら洗い出せる
  → 会員ステータス・購入金額・支払い方法を網羅的に導出できる
```

テスト分析においては、**意図的に抽象度を保つことがむしろ品質につながる**。

POから具体的すぎる条件が渡ってきたら、上流に戻って仕様の抜け漏れを確認することもテスト分析の重要な仕事です。

## テスト設計：「どのようにテストするか」

### カバレッジアイテムとは

> 「1つ以上のテスト条件から導出される属性または属性の組み合わせ」

テスト技法（同値分割・境界値分析・デシジョンテーブルなど）を使って、
カバレッジ基準を満たす具体的な網羅の粒を識別します。

### 同値分割

属性のバリエーションを「同じ振る舞いをするグループ（パーティション）」に分ける技法。

```
購入金額という属性
  ├─ 有効パーティション：5000円未満 / 5000円以上
  └─ 無効パーティション：負の値 / 数値以外
```

**有効パーティション**：システムが処理すべき値のグループ
**無効パーティション**：システムが拒否すべき値のグループ

無効パーティションをテストするかどうかも、リスク分析の結果として決まります。
システム的に無効値の入力が不可能な場合（UIで制御済み・API検証済み）は除外を検討します。

### 境界値分析

パーティションの境界を代表値として選ぶ技法。欠陥は境界に集中しやすい。

```
「5000円以上は無料」という仕様の場合
  └─ 4999円と5000円の処理を間違えるバグが最も起きやすい
  └─ パーティション内の代表値（3000円）だけでは検出できない
```

境界値分析は「同値分割をより欠陥検出に特化させた拡張」と捉えるのが自然です。

### カバレッジ基準の種類（複数属性がある場合）

複数の属性がある場合、組み合わせ方針がカバレッジ基準になります。

| カバレッジ基準 | 内容 | テストケース数の目安 |
|---|---|---|
| イーチチョイス | 各パーティションが最低1回テストされる | パーティション数の最大値 |
| ペアワイズ | 任意の2属性の組み合わせを全網羅 | 全組み合わせより大幅削減 |
| 全組み合わせ | 全パーティションの組み合わせを網羅 | 各属性のパーティション数の積 |

## 具体例：ECサイトの送料計算

### 仕様

- 会員ステータス：一般 / プレミアム
- 購入金額：5000円未満 → 一般500円・プレミアム250円 / 5000円以上 → 無料
- 支払い方法：クレジットカード / 代引き（代引きは+330円）

### テスト目的・リスク・カバレッジ基準

```
コンテキスト：決済系・本番リリース前

テスト目的
  ├─ 欠陥の検出
  └─ 要件を満たしていることの確認

リスク
  └─ 送料計算ミスは直接金銭的損害につながる → 高

カバレッジ基準
  └─ 全組み合わせ100% + 境界値必須 + 無効パーティション含む
```

### カバレッジアイテム（有効パーティション：8件）

| # | 会員 | 購入金額 | 支払い | 期待送料 |
|---|---|---|---|---|
| 1 | 一般 | 5000円未満 | クレカ | 500円 |
| 2 | 一般 | 5000円未満 | 代引き | 830円 |
| 3 | 一般 | 5000円以上 | クレカ | 無料 |
| 4 | 一般 | 5000円以上 | 代引き | 330円 |
| 5 | プレミアム | 5000円未満 | クレカ | 250円 |
| 6 | プレミアム | 5000円未満 | 代引き | 580円 |
| 7 | プレミアム | 5000円以上 | クレカ | 無料 |
| 8 | プレミアム | 5000円以上 | 代引き | 330円 |

### 境界値（4件）

| # | 属性 | 入力値 | 期待結果 |
|---|---|---|---|
| 9 | 購入金額 | 4999円 | 500円 |
| 10 | 購入金額 | 5000円 | 無料 |
| 11 | 購入金額 | 0円 | 500円 |
| 12 | 購入金額 | -1円 | エラー |

## 業務での課題と気づき

### 「バグゼロ＋時間的制約」という矛盾

よくあるパターン：

```
PO：「バグゼロで！でも時間は限られてるからよしなに」

本来の意味：
  バグゼロ → テスト目的は欠陥検出・全リスク対応
           → 全テストレベルで全組み合わせ100%
           → カバレッジアイテムが膨大

よしなに → テストケースを減らす
         → でも何を根拠に減らすの？
```

これは**テスト目的・リスク・カバレッジ基準が定義されていない**ために起きます。

### 解決策：数字と根拠で話す

```
✅「この機能はリスク高のため全組み合わせ8ケース必要です」
✅「リスク中に下げるなら境界値のみで4ケースに絞れます」
✅「イーチチョイスにするならXケースで済みますが、
   この組み合わせの欠陥は検出できません」
```

目的・リスク・基準を明確にすることで、**テストの範囲を根拠を持って説明できる**ようになります。

## 作ったもの：テストケース生成スキル

### 課題

- PBIチケットとコードはあるが、既存仕様書がない
- テスト目的・リスク・カバレッジ基準を毎回考えるのは大変
- でもこの一連の流れを省略するとPOを説得できない

### 解決策

PBIチケット + 既存コードをインプットに、以下を自動化するClaudeスキルを作成。

```
STEP1: コードから仕様を把握（テストベース分析）
STEP2: テスト目的・リスクを提案 ← 開発者との合意ポイント
STEP3: カバレッジ基準を決定
STEP4: カバレッジアイテム（テストケース）をMarkdownテーブルで生成
```

### アウトプット例

```markdown
## テストケースサマリー
- テスト対象：送料計算（calcShipping）
- テストレベル：単体テスト
- カバレッジ基準：全組み合わせ × 有効+無効
- テストケース総数：14件

| # | member | amount | payment | 入力値例 | 期待結果 |
|---|---|---|---|---|---|
| 1 | general | 5000円未満 | card | 3000 | 500 |
...
```

## まとめ

### テスト設計の流れ

```
テスト計画
  └─ コンテキストを踏まえてテスト目的・リスクを定義
       ↓
テスト分析
  └─ テスト条件を抽出
  └─ カバレッジ基準を設定（何をもって100%とするか）
       ↓
テスト設計
  └─ テスト技法でカバレッジアイテムを導出
  └─ テストケースへ落とし込む
```

### ゴールの達成

> AIに指示が出せない・何を実装すればいいか分からない

この状態の根本原因は「**実装対象の振る舞いが整理できていない**」こと。

テスト設計の考え方を身につけることで：

```
PBIチケット + コード
  ↓ テストベース分析
振る舞いが整理される（テスト条件・カバレッジアイテム）
  ↓
「この入力のとき、この出力になるべき」が明確になる
  ↓
AIへの指示が具体的に出せる
```

**テストは実装後のものではなく、実装前に振る舞いを整理するための思考ツールでもある。**

### memo

#### リスク
以下の二つの特性を持つ。発生可能性がある不利益をもたらす「何か」
1. 不確実性：リスクを有する事象は起きたり起きなかったりする。すなわち100%発生するリスクというものはない
2. 損失：リスクが現実となった場合は、望まれていない結果、または損失が発生する。

→ リスクはリスクの可能性と現実となった時にどの程度不利益をもたらすかを考慮した上で識別し、リスクレベルを決める
