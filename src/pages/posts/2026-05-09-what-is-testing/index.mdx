## テストを学ぼうとしたきっかけ

### 物量勉強法の限界を直感した

新卒で最初に入った案件では、触ったことのない技術や領域を実務レベルで一気に身につける必要があり、かなり挫折しかけていました。

そんな時、これまでみたいに物量で押し切る勉強法では無理だと直感し、個別の技術を一つずつ追うよりまずはいろんな技術に共通する幹を掴んだほうがいいと切り替え、テストを学び始めました。

### AIに指示が出せなかった

個人で勉強したり開発したりしているときはそれなりにコードが書けていたのに、なぜか案件に入ってから全く手が動かない。
だったらAIに指示を出そうと思うけど、なんて指示をすればいいのかもわからない！

そんなことが続く中で、そもそも自分の中でどうなっていたら良いのかという理想の状態が分からない。だから人にもAIにも尋ねようがないと気が付きました。

同じような悩みが実装だけでなく、高頻度でテスト設計でも起きることに気づきテストを学べば俺は何をすればいいのかの手がかりがあるのでは？
と当時すがるような思いでテストについて学ぼうとしていました。

※こねこね頑張ってなんとか2025年の4月にJSTQB Foudation Levelに合格できました。

## テストの目的

テスト活動の話をするにあたって、私たちがテストをする目的ってなんでしょうか？
バグゼロを目指すことでしょうか？（テストで欠陥があることは示せても、欠陥がないことは示せないのでこれは難しいですね）

ISTQBでは複数定義されており、代表的なものを挙げると以下のようになります。

| テストの目的 | 内容 |
|---|---|
| 欠陥の検出 | 故障を引き起こし、ソフトウェアの欠陥を発見する |
| 要件の確認 | 実装が要件・仕様を満たしていることを確認する |
| カバレッジの確保 | 重要な機能や特性を見逃さないようにする |
| リスクの低減 | 本番障害のリスクを事前に下げる |
| … | 品質への信頼の構築・法規制への準拠など |

重要なのは、**目的はコンテキストによって変わる**という点です。

```
納期が迫っている    → 必要最低限の機能が動くことの確認が目的
リリース前の決済系  → 欠陥の検出・要件確認が目的
機能追加後         → 既存機能への影響がないことの確認が目的
```
<br />

この「テストの目的」はテスト活動をする上でまず最初に決める必要があります。
なぜ最初にテスト目的を決める必要があるかまではシラバスにも教科書にも明確に書かれていませんが、目的がないのに動き出そうにも動け出せませんよね。

「鷹村守」のようななんでもできるエンジニアになりたい！という目的があるから日々コツコツと、私はいろんな分野の勉強をしています。目的もないのに勉強できる人・目的もないのに動ける人というのはなかなかいないと思います。

とはいえテスト目的という指標なくテスト活動を進めるもしくは、テスト目的という概念を知覚していない状態でソフトウェア開発が進んでいくことはあると思います。
そうなると経験上、**終わりのないテストの始まり**になります。

ハンター試験の1次試験のように、ゴールが見えないというのは非常にストレスがかかります。
ストレスと疲弊が積み重なり、最終的には「よしなに調整する」という判断となるでしょう。

よしなに調整された基準(≒テスト目的）がここでハッキリ明文化すれば、今後のテスト活動は改善する可能性がありますが、
この概念を知らなければそのような発想は難しいだろうなと私は思います。

そのためよしなな調整というのはその人本人の中でうっすらとしたものとして残り、
属人的なテスト活動が進むことでメンバーによって品質のばらつきが出始めると思います。
（とはいえ属人化自体は完全には避けられないですし属人化という経験を元にしたテスト技法も存在しますが）

あと目的は誰が見ても達成できているなと思えるような定量的なものが好ましいと思います。さっきの鷹村守（なんでもできる）のようなエンジニアとなったかどうかをどうやって判断するかの指標はないのでよくない目的ですね笑




## テストプロセスの全体像（テスト計画からテスト実装までに限る）

テスト目的が決まりいざテストケース作るぞ！となるくらいテスト目的が具体な場合もあれば、機能が正しく動くことみたいなハイレベルなこともあると思います。

テスト活動ではテスト目的を決めてからテストケースの導出まで、2ステップほど踏みます。（ほどというのはコンテキストに応じてテスト分析を飛ばしてテスト設計に入ることもあるからです）

シラバスには他にもテストフェーズはあるのですが、今回はテストケースの作成までを目標には話したいので、テスト分析・テスト設計までを辿っていきます

```
テスト計画
  └─ テスト目的・リスクを定義

テスト分析
  └─「何をテストするか」を明らかにする
       ├─ テスト条件の抽出
       └─ カバレッジ基準の設定

テスト設計
  └─「どのようにテストするか」を明らかにする
       ├─ カバレッジアイテムの導出
       └─ テストケースの設計

テスト実装 → テスト実行
```

### なぜプロセスを踏むのか

プロセスに沿って進む理由は主に2つあります。

**1. テストすべき範囲を絞れる**

目的を定義し、分析でテスト条件を抽出することで、「やらなくていいこと」が明確になります。全部テストしようとするのではなく、コンテキストとリスクに応じてスコープを絞っていきましょう。

**2. 行き当たりばったりのテストでなくなる**

プロセスを踏まないと、何となく思いついた観点だけテストして終わる可能性があります。計画・分析・設計というステップを踏むことで、「なぜこのテストケースが必要か」に根拠が生まれ、漏れや格段に減り、もっとやるべきテストあったのに時間切れでできなかった〜なんてことを防げます。

以降はテスト分析・テスト設計を中心に掘り下げます。



## テスト分析：「何をテストするか」

### テスト条件とは

> 「テストによって確認できるもの/こと」

ポイントは**抽象度に幅がある**ことです（これむずい）。
またテスト条件は、PBIのチケットや仕様書、プロダクトコードといったテストベースから情報を取捨選択していきながら決めていきます。

| 抽象度 | テストベース | 例 |
|---|---|---|
| ハイレベル | PBIチケット・要件定義 | 「送料計算が正しく機能すること」 |
| 中レベル | 仕様書・設計書 | 「会員ステータスによって送料が変わること」 |
| 詳細 | プロダクトコード・詳細仕様 | 「通常会員・5000円未満・クレカ支払い時、送料が500円となること」 |


### カバレッジ基準


会員ステータスがプレミアム会員の場合だけテストすればいいのでしょうか？他にもステータスが存在していて、それもテストするべきなのでしょうか？

このようにテスト条件が決まっただけでは、どこまでやれば完了かがまだ曖昧ですね


カバレッジ基準（**何をもって100%とするかのルール**）を設定することで、「完了 = どこまで何をテストするのか」を定義していきます。

つまり私の理解だとテスト条件が何をテストするのかというスコープで、さらにカバレッジ基準でどこまでやるのかを絞っていきます。

なので、シラバスや教本の中では、テスト分析では「何をテストするのか？」という問いに答えているとありますが、
テスト分析とは **「どこまで」何をテストするのか？** という問いに答えている と表現の方が私はしっくりきます。

```bash
カバレッジ基準の例
  ├─「送料計算に影響する属性をすべて対象にする」
  ├─「会員 × 購入金額 × 支払い方法の組み合わせを確認する」
  └─「購入金額は4999円と5000円を必ず確認する」
```
<br/>

カバレッジ基準はテスト対象のもつリスクと連動します。リスクが高いほど基準が厳しくなり、テストケースの数が増えます。コンテキスト次第というやつですね！

### テスト条件の抽象度に幅があることは大切

具体的すぎるテスト条件が渡ってきた、見つけた場合にはちょっとしたリスクがあると私は考えています

```
❌「通常会員の場合、正しく送料計算されること」
  → 会員ステータスの観点しか見えない
  → 購入金額・支払い方法の観点が抜ける危険

✅「送料計算が正しく機能すること」
  → 送料に影響する属性を自ら洗い出せる
  → 会員ステータス・購入金額・支払い方法を網羅的に導出できる
```
<br/>

テスト分析においては、**意図的に抽象度に幅を持たせることがむしろ品質につながるのではないかと思います**。

具体的すぎる条件が渡ってきたら、上流に戻って仕様の抜け漏れを確認することもテスト分析の重要な仕事です。
そのため、テスト分析は「仕様書自体の欠陥を見つける」フェーズでもあります。シフトレフトですね！

## テスト設計：「どのようにテストするか」

テスト分析で「何をテストするか」と「何をもって100%とするか（カバレッジ基準）」が決まりました。
 
テスト設計ではその続きとして、**その条件と基準をどうやって具体的なテストケースに落とし込むか**を考えます。
 
シラバスでは「どのようにテストするか」という問いに答えるフェーズと説明されていますが、
ここでいう「どのように」とは、より具体的には**どの組み合わせを、どの値で、どの期待結果として確認するかを決めること**だと私は考えています。
 
```
テスト分析
  └─ 何をテストするか
       └─ テスト条件の抽出 → カバレッジ基準の設定
 
テスト設計
  └─ どのようにテストするか
       ├─ 属性を洗い出す
       ├─ 組み合わせを整理する
       ├─ 具体的なテストデータを決める（4999円・5000円など）
       └─ テストケースとして出力する
```
 
 
### カバレッジアイテムを導出するには
 
 
「送料計算の全パターンを網羅する」というカバレッジ基準があったとして、
どんな値を使ったカバレッジアイテムを用意すればそれを満たせるかを感覚で決めると漏れが出ます。

そこで登場するのがテスト技法です。

次のように捉えるとかなり整理できるかと思います

1. **同値分割**で、テスト条件（振る舞い）に影響する属性とパーティションを識別する
2. **デシジョンテーブル**で、その属性の組み合わせを整理する
3. **境界値分析**で、各パーティションにどの具体値を入れるかを決める
4. その結果、**テスト条件に対する具体的なテストケース**が定まる

つまり、3つの技法はそれぞれ独立しているというより、**「何をテストするか」を「どのようにテストするか」へ変換するための連続したステップ**として見ると良いかもしれないですね！
テスト技法を使うことで、**漏れなく・根拠を持って**カバレッジアイテムとテストデータを導出できます。

### 同値分割
 
同値分割を使うには、まずテスト条件から「属性」を洗い出します。属性とは、テスト条件に影響を与える要素のことです。

たとえば、送料計算に次のような仕様があるとします。

- 通常会員は送料500円
- プレミアム会員は送料無料
- 購入金額が5,000円以上なら会員種別に関係なく送料無料
- 代引きは追加で手数料300円かかる

この仕様に対して、ハイレベルなテスト条件としては `送料計算が正しく機能すること` と置けます。

ただ、このままではまだテストケースは作れません。そこで、送料計算に影響しそうな属性を洗い出します。
 
```
テスト条件
  └─「送料計算が正しく機能すること」
       ↓ 属性を洗い出す
  ├─ 会員ステータス
  ├─ 購入金額
  └─ 支払い方法
```

<br/>

ここでの属性は `会員ステータス` `購入金額` `支払い方法` です。コンテキストによっては、配送地域や日時指定の有無のような属性が増えるかもしれません。

そしてその属性ごとに、「同じ振る舞いをするグループ（パーティション）」に分けていくのが同値分割という技法です。
 
```
購入金額という属性
  ├─ 有効パーティション
  │    ├─ パーティションA：5000円未満
  │    └─ パーティションB：5000円以上
  └─ 無効パーティション
       ├─ パーティションC：負の値
       └─ パーティションD：数値以外
```

<br /> 

**有効パーティション**はシステムが処理すべき値のグループ、**無効パーティション**はシステムが拒否すべき値のグループです。
 
同じグループ内であれば代表値が1つあれば十分、という考え方がこの技法の肝です。
5000円未満であれば1円でも3000円でも同じ振る舞いをするはずなので、代表値を1つ選んでテストすれば足ります。
 
無効パーティションをテストするかどうかはリスク分析の結果として決まります。

UIで会員ステータスが選択式になっており、システム的に不正な値を渡せない設計になっているなら、
その属性の無効パーティションは除外を検討してよいと思います。
 
### デシジョンテーブル

同値分割では属性ごとにパーティションを分けましたが、実際の仕様は複数の属性が組み合わさって決まることが多いです。送料計算の仕様をおさらいすると：

| | 5000円未満 | 5000円以上 |
|---|---|---|
| 通常会員 | 500円 | 無料 |
| プレミアム会員 | 無料 | 無料 |

代引きの場合は上記に +300円です。

属性のパーティションを整理すると、会員（2） × 購入金額（2） × 支払い方法（2）で **8通りの組み合わせ**があることがわかります。これを頭の中だけで漏れなく把握しようとするのはなかなか大変です。そこで登場するのがデシジョンテーブルです。

![送料計算のデシジョンテーブル](/posts/2026-05-09-what-is-testing/decision_table.svg)

こうして表に起こすことで、「この組み合わせ、考えてなかった」という抜け漏れを防げます。
また「なぜこの8ケースなのか」という根拠も明確になります。

この表の1列（例：「通常会員 × 5000円未満 × クレカ → 500円」）が、**カバレッジ基準を満たすためのカバレッジアイテム**の1つです。
まだこの段階では「どのパターンを確認するか」が決まっただけで、具体的にどの値（例2500円、200円など）を入れるかまでは確定していません。

### 境界値分析

デシジョンテーブルで「どの組み合わせを確認するか」が整理できたら、次はその組み合わせに**どの具体値を入れるか**を決める必要があります。
これがテストデータの識別であり、境界値分析はその役割を担う技法です。
 
欠陥は境界に集中しやすいという経験則に基づき、パーティションの境界値を代表として選びます。

今回の例では、境界値分析を特に効かせたいのは購入金額です。会員ステータスや支払い方法は、この仕様では代表値を選べば十分です。
 
```
「5000円以上は無料」という仕様の場合
  └─ 4999円と5000円の処理を間違えるバグが最も起きやすい
  └─ パーティション内の代表値（3000円）だけでは検出できない
```
<br/>

テストの目的の一つが「欠陥を見つける」ことであるなら、欠陥が潜みやすい場所を狙いに行く必要があります。

パーティション内の適当な値だけ確認して終わるのは「振る舞いを確認した」に過ぎず、
「欠陥を狙いに行った」とは言えないですよね。
 
つまり、

- **同値分割**で属性とパーティションを識別する
- **デシジョンテーブル**で組み合わせを整理する
- **境界値分析**で具体的なテストデータを決める

という流れを経て、はじめて「どの組み合わせに、どの値を入れて、どう確認するか」が決まり、
テストケースとして出力できるようになります。

 
 
### カバレッジ基準の種類
 
複数の属性がある場合、組み合わせをどこまで網羅するかがカバレッジ基準になります。代表例を2つ紹介します。

- **イーチチョイス**：各パーティションが最低1回テストされればOK。会員（2）・購入金額（2）・支払い（2）の3属性なら最低2ケースで済みます。
- **全組み合わせ**：全パーティションの組み合わせを網羅。2×2×2＝8ケース必要です。

どちらが正解というわけではなく、**リスクとコンテキストに応じて使い分けます。**

他にも存在しますが、Foundation Levelでは紹介されていないので、詳しく知りたい方は好奇心の赴くまま調べてみてください。

## リスク分析

テストの優先順位つける上で大切な概念がリスクです。
リスクの定義は様々あって混乱しますが、一番私がしっくりきているのは「リスクとは発生可能性がある不利益をもたらす何か」です。

またリスクは様々定義がありますが共通した2つ性質を持っています。

1. 不確実性：リスクを有する事象は起きたり起きなかったりする。すなわち100%発生するリスクというものはない
2. 損失：リスクが現実となった場合は、望まれない結果、または損失が発生する。

この性質からリスクを定量的に数値化したものが、リスクレジスターです。

### リスクレジスター

リスクをP・C・Eの3指標で数値化します。

```
P（Probability）= 発生確率（1〜6）
C（Consequence）= 影響度（1〜6）
E（Exposure）   = P × C（最小1・最大36）
```

| リスクID | リスク内容 | P | C | E | 緩和活動 |
|---|---|---|---|---|---|
| 1 | 境界値（4999/5000円）で送料が誤っている | 3 | 5 | 15 | 境界値テスト必須・自動テスト化 |
| 2 | 組み合わせによって誤った送料になる | 3 | 6 | 18 | デシジョンテーブルで全組み合わせ |
| 3 | 代引き手数料の加算が誤っている | 2 | 4 | 8 | 単体テスト・コードレビュー |
| 4 | 無効値で例外が発生しない | 1 | 3 | 3 | 無効パーティションテスト |

E（Exposure）が大きいものがリスクが大きく優先度高めのテスト対象だ！だからカバレッジ基準を厳しくしよう。
と判断できるようになり、そうしてテストケースが導出できますね！



## 業務での課題と気づき

### 「バグゼロ」を掲げるとむしろ品質が下がるのでは？



```
「バグゼロ」宣言
  └─ 全リスクが同等に高い扱いになる
  └─ リスクの差別化ができない
  └─ 相対的な優先順位が決められない

       ↓

結果として
  ├─ 全部やろうとして時間が足りない
  ├─ 時間が足りないから、リスク激高のテスト対象をテストできなかったなんてことが起こりうる
  └─ よしなにやってください→テスターの経験次第で品質がブレる
```
<br/>



### 「バグゼロ＋時間的制約」という矛盾にどう対応するか

リスク分析結果という根拠を持って、優先度を決める判断をAIと協働して行う必要があると思います。

```
✅「リスクID2はE=18で最優先。全組み合わせ8ケーステストします！」
✅「リスクID4はE=3。時間があればやります！」
✅「時間が足りない場合、E=8以下を後回しにする
   というトレードオフを提案する」
```
<br/>

AIと協働するということで↑のテストフェーズをベースに、リスクスコアを元に優先度を判断し、
テストケースを作成するスキルを作ってみました

## 作ったもの：テストケース生成スキル

<LinkCard url="https://github.com/8mitsuboy/skills/blob/main/test-case-generator/SKILL.md" />

### 課題

- PBIチケットとコードはあるが、既存仕様書がない
- テスト目的・リスク・カバレッジ基準を毎回考えるのは大変
- でも根拠あるテストケースを作りたい！


PBIチケット + 既存コード + その他テストベースをインプットに、以下を自動化するClaudeスキルを作成してみました。

```
STEP1: コンテキストを対話的に収集（Q1〜Q3に答えるだけ）
STEP2: コードから仕様を把握（テストベース分析）
STEP3: リスクレジスターを作成・開発者と合意
STEP4: Eスコアとコンテキストからカバレッジ基準を決定
STEP5: カバレッジアイテムをMarkdownテーブルで生成
```
<br />

STEP1のコンテキスト収集では、以下のような質問が飛んできます。
 
```
Q1. リリース時期はいつ頃ですか？
  A. リリース直前（1週間以内）
  B. 開発中（リリースまで余裕あり）
  C. わからない
 
Q2. 機能の種類はどれに近いですか？
  A. 決済・金融・個人情報など重要機能
  B. 通常の業務機能
  C. UI調整・軽微な変更・内部リファクタ
 
Q3. 無効な入力値はシステム的に防がれていますか？
  A. UIや上流のバリデーションで防がれている
  B. 防がれていない（APIを直接叩ける・入力が自由）
  C. わからない
```
<br/>

この回答をもとに、無効パーティションを含めるかどうかやカバレッジ基準の厳しさが自動的に決まります。
 
### アウトプット例
 
```markdown
## テストケースサマリー
- テスト対象：送料計算（calcShipping）
- テスト目的：欠陥検出・要件確認
- 最高Eスコア：18（組み合わせによる送料誤り）
- カバレッジ基準：全組み合わせ × 有効+無効 × 境界値必須
- テストケース総数：12件
| # | member | amount | payment | 入力値例 | 期待結果 |
|---|---|---|---|---|---|
| 1 | general | 5000円未満 | card | 3000 | 500 |
...
```

## まとめ

### テスト設計の全体像

```
テスト計画
  └─ コンテキストを踏まえてテスト目的・リスクを定義
       ↓
テスト分析
  Input:
    - PBIチケット
    - 要件・仕様
    - 既存コード
    - 設計情報
  Behavior:
    - テスト条件を抽出する
    - リスクを洗い出してリスクレベルを決める
    - カバレッジ基準を設定する
  Output:
    - テスト条件
    - リスクレジスター
    - カバレッジ基準
       ↓
テスト設計
  Input:
    - テスト条件
    - リスクレジスター
    - カバレッジ基準
  Behavior:
    - テスト技法でカバレッジアイテムを導出する
    - テストデータを決める
    - テストケースへ落とし込む
  Output:
    - カバレッジアイテム
    - テストデータ
    - テストケース
```

### テスト設計と振る舞いの整理

「AIに指示が出せない・何を実装すればいいか分からない」の根本原因は「**実装対象の振る舞いが整理できていない**」から。

テスト設計の考え方を身につけることで：

```
PBIチケット + コード
  ↓ テストベース分析
振る舞いが整理される（テスト条件・カバレッジアイテム）
  ↓
「この入力のとき、この出力になるべき」が明確になる
  ↓
AIへの指示が出せる（AIへ適切なゴールを指定できる！）
```

<br/>

テストを勉強して振る舞いという概念を掴めたし、大きいところから小さく刻んでいくステップは、
複雑なタスクをする上でも重要な考え方で、Next.jsのメジャーアップデートの時にタスク迷子にならずに完走できた要因だな〜と思うのであった

テスト奥深すぎてなんもわからん。
