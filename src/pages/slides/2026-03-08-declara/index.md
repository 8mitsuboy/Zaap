# 私の考える先延ばしの正体（怠惰ループ）

### 例.「引っ越したい」
<br />

- 1. やるべきことがわからない。順序がわからない。（主に外在的負荷）
- 2. 面倒になって後回しにする
- 3. 何も進んでなくて落ち込む
- 4. 落ち込んで現実逃避 →1へループ

<br />

### 俺は引っ越したい（新居で暮らしている状態になりたい）だけで <br /> 引越し業者の手配とか住所変更とかしたくない！

---

# 悪循環の起点

**何をすればいいかわからない = ブラックボックス**

わからない → 面倒 → 落ち込む → 動けない → わからない

**怠惰が悪いのではない。ブラックボックスが悪い。**

---

# 既存Todoのつらさ

「引っ越したい」を入力する前に

1. タスクを分解する
2. 順番を考える
3. 抜け漏れを心配する

**初動でHowを要求される**

---

# 発想の転換

```txt
手続き的: Howを全部書く
宣言的: Whatだけ書く
```

タスク管理でも同じにする。

- ユーザー: Whatを宣言
- AI: Howを生成

---

# Declara

「宣言するだけ」のタスク管理アプリ

- 入力は「〜したい」「〜しなきゃ」だけ
- AIが中身を開封してタスク化
- 日常の後回しを片付ける用途に特化

---

# 感情フローを設計対象にした

| フェーズ | 感情 | UI方針 |
|---------|------|--------|
| 停滞 | 何すれば… | 出発点として受け止める |
| 宣言 | ポンと入力 | 入力UIを極小化 |
| 開封 | これだけでいい | タスクを見やすく提示 |
| 実行 | 進んでる | 進捗可視化 + 完了演出 |
| 達成 | やりきった | 小さく祝う |

---

# MVPの機能要求（抜粋）

- 宣言入力 → タスク生成
- 生成タスク数は **3〜7件** に制御
- 空入力は即エラー
- 生成失敗時は再試行エラー
- タスク完了状態を永続化
- 進捗（完了数/総数）を表示

---

# MVPの非機能要求（抜粋）

- 生成開始から結果表示まで **3秒以内目標**
- 例外時でもクラッシュしない
- 生成ロジックはUIから分離
- 将来LLM差し替え可能な構成

---

# 実装

- Flutter + Riverpod
- Drift (SQLite) で永続化
- Claude API でタスク生成
- DDDで層分離（Presentation / Domain / Data）

---

# デモで見るポイント

「引っ越したい」と宣言するだけで

1. 3〜7件の具体タスクが出る
2. 1つ完了すると進捗が動く
3. 全完了時に達成感を返す

# 学び

- 先延ばしの敵は意志の弱さではなく情報の不透明さ
- 宣言的プログラミングはUX設計にも使える
- 生成AIの価値は「考える負担を消す」こと

---

# もっと突き詰めると

「引っ越したい」= 「引っ越した状態になりたい」

これは**理想の状態の宣言**。Reactの `<Button>` と同じ。

ユーザーは宣言していることすら意識していない。

---

# でも現実の壁

「理想の状態を宣言して」と言った瞬間…

- 「幸せになりたい」→ 抽象的すぎて分解できない
- 「食器を洗いたい」→ 具体的すぎて分解不要
- 「引っ越したい」→ **ここだけが刺さる**

独自性を出すとユースケースが狭まる。汎用的にすると埋もれる。

---

# プロダクト設計の難しさ

コードはコンパイラが正解を教えてくれる。

でもプロダクト設計には正解がない。

**コードを書くより、何を作るかを決める方が難しかった。**
